[서비스 이름] 주말 독서 토론 참가 관리 서비스

[한 줄 소개]
주말의 독서 토론 소개하고, 손님이 예약 문의를 남기면
사장이 관리자 화면에서 접수 내역을 확인·확정·삭제할 수 있는 서비스

[핵심 기능 — 5가지]
1. 서비스 소개: 클래스 설명, 가격, 위치를 보여준다
2. 예약 문의 접수: 이름·연락처·희망 날짜·요청사항을 입력받아 저장한다 (Create)
3. 관리자 조회: 사장만 로그인해서 접수 내역을 최신순으로 본다 (Read)
4. 접수 상태 변경: 사장이 "대기 → 확정"으로 상태를 바꾼다 (Update)
5. 접수 삭제: 사장이 스팸·취소 접수를 삭제한다 (Delete)

[화면 구성]
■ 고객 페이지 (index.html)
- 상단: 대표 문구 "혼자 읽던 책, 이제 함께 이야기해요 — 주말 독서 토론" + 서비스 한 줄 소개 (히어로 영역)
- 중간: 소개 카드 3개 (무엇을 / 얼마에 / 어디서)
- 하단: 예약 문의 폼 (이름, 연락처, 희망 날짜, 요청사항, 접수 버튼)

■ 관리자 화면 (admin.html) 
- 로그인 폼 (이메일, 비밀번호)
- 로그인 후: 접수 내역 표 + 각 행에 [확정] [삭제] 버튼
- 상태별 표시: 대기(회색) / 확정(초록)
- 로그아웃 버튼

[데이터 — 저장할 것]
- 접수 ID (고유 식별자, 자동)
- 이름 (텍스트) / 연락처 (텍스트) / 희망 날짜 (날짜)
- 요청사항 (텍스트) / 상태 (텍스트, 기본값 "대기") / 접수 시각 (자동)

[권한 설계 — 누가 무엇을 할 수 있나]
- 손님(로그인 없음): 접수 생성(Create)만 가능. 조회·수정·삭제는 불가
- 사장(로그인함): 조회(Read)·상태 변경(Update)·삭제(Delete) 가능. 생성(Create)은 범위 아님
- 권한 구분은 화면(UI)에서 로그인 폼을 숨기는 것만으로 끝나지 않는다. 실제 데이터 접근 제어는 DB 정책(Supabase RLS)으로 강제한다:
  - INSERT(접수 생성): 로그인 없이도 허용 (손님용)
  - SELECT·UPDATE·DELETE(조회·상태변경·삭제): 로그인한 사장 계정만 허용, 그 외에는 전부 거부
  - 즉 admin.html에 로그인하지 않은 상태로 접속하거나 API를 직접 호출해도 접수자 이름·연락처는 조회되지 않아야 한다
- 관리자 계정은 Supabase Auth에 1개만 존재하며, 앱 어디에도 회원가입(sign-up) 경로를 만들지 않는다 (화면뿐 아니라 기능 자체를 배제)

[만들지 않는 것 (범위 제외)]
- 고객 회원가입 없음 (손님은 로그인 없이 접수, 변경 요청은 사장에게 연락)
- 관리자 "가입" 기능 없음 (관리자 계정 1개는 Supabase 화면에서 직접 생성)
- 결제 기능 없음
